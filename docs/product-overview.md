# CDK Express Mode 検証プロダクト概要

## 背景

2026 年 6 月末、AWS は **CloudFormation and CDK Express Mode** をリリースした。
このモードは、スタック操作の完了判定を「リソース設定の適用が確認できた時点」に前倒しする。

従来の CloudFormation は、リソース作成後に以下のような安定化チェックの完了まで待ってから
デプロイ完了と判定していた。

- トラフィック準備（traffic readiness）
- リージョン伝播（region propagation）
- リソースのクリーンアップ（resource cleanup）

Express Mode ではこれらの処理をバックグラウンドで継続させ、設定適用時点でデプロイを完了扱いにする。
その結果、デプロイ時間が最大 4 倍高速化するとされている。

## 目的

本プロダクトは、Express Mode の効果を **実測で定量的に検証する** ことを目的とする。

具体的には、安定化・伝播に時間がかかるリソースを含む CDK スタックを用意し、
通常デプロイと Express デプロイの所要時間を比較する。

### 検証スコープ

**本検証で扱うのは CDK での有効化（`cdk deploy --express`）のみ** とする。
AWS CLI（`--deployment-config`）やマネジメントコンソールからの有効化は検証対象外。

理由は、本プロジェクトが CDK ベースの開発フローにおける Express Mode の効果測定を目的としているため。
CLI／コンソールの挙動は仕様上同等とされるが、実測は CDK 経由に絞って条件を単純化する。

あわせて、以下の実運用上の観点も確認する。

- 高速化の効果が大きいリソース種別はどれか
- Express Mode でデプロイ完了と判定された「後」に、バックグラウンド処理が本当に続いているか
- ロールバックがデフォルト無効であることによる挙動（失敗時に中途半端な状態が残ること）
- 既存テンプレートを変更せずにモードだけ切り替えられるか

なお、`--express` と `--rollback` を組み合わせたロールバック再有効化の挙動は **検証対象外** とする
（公式ドキュメント未記載であり、本検証の主眼であるデプロイ時間の比較から外れるため）。

## 検証対象アップデートの要点

| 項目 | 内容 |
|------|------|
| 機能名 | CloudFormation and CDK Express Mode |
| 効果 | デプロイ時間を最大 4 倍短縮 |
| 完了判定 | リソース設定の適用完了時点（安定化チェックは背景で継続） |
| CDK での有効化 | `cdk deploy --express` |
| CLI での有効化 | `--deployment-config '{"mode": "EXPRESS"}'` |
| ロールバック | デフォルト無効（`disableRollback: false` で再有効化可能） |
| テンプレート変更 | 不要 |
| 対応範囲 | 既存テンプレートと互換。ネストスタック・Change Sets にも対応 |
| 対応リージョン | CloudFormation が利用可能な全リージョン |

公式が挙げている効果例は次のとおり。

- SQS + DLQ: 通常 64 秒 → Express 最大 10 秒
- VPC 内 Lambda（ENI 付き）の削除: 通常 20〜30 分 → Express 最大 10 秒

### ドキュメント上の注意点（要検証）

調査時点で、以下は公式ドキュメントに明記されていない。実装・検証時に実際の挙動を確認する。

- CDK CLI の `cdk deploy` コマンドリファレンスに `--express` オプションはまだ掲載されていない
  （`--express` は What's New とブログでのみ言及）。
- Express モードでロールバックを再有効化する公式手順は、CLI/API の `disableRollback: false` のみ。
  CDK での `--express --rollback` という組み合わせの挙動は未記載。
- 既存の `--rollback` / `--no-rollback` は Express とは独立した標準オプション。
  `--express` がこれらとどう相互作用するか（デフォルトを上書きできるか等）は未記載だが、本検証では扱わない。

## 検証で確かめたい仮説

1. どれぐらい高速化するかをリソースの種類ごとと、トータルの両方で計測
2. 安定化・伝播が重いリソースほど、Express Mode の時短効果が大きい。
3. Express Mode の完了判定は速いが、実際のリソースが「使える状態」になるまでには
   バックグラウンド処理の時間が別途かかる場合がある。
4. 同一テンプレート（CDK コード）のまま、フラグだけでモードを切り替えられる。
5. Express Mode ではロールバックが無効なため、失敗時は中途半端な状態で残る。

## 検証スタックの構成

CDK（TypeScript）で、安定化・伝播に時間がかかるリソースを意図的に積んだスタックを構築する。
リソースは役割ごとに「軽量セット」と「重量セット」として整理するが、**両方を 1 つのスタックにまとめてよい**。
複数スタックに分ける場合も、`cdk deploy --all` で **1 回のデプロイ** にまとめる。

### 軽量セット（低コスト・差分が明確）

- SQS キュー + デッドレターキュー（DLQ）
- VPC + VPC 内 Lambda（ENI 作成・削除が遅い代表例）
- SNS トピック / SSM パラメータ など補助リソース

### 重量セット（一般的な Web サービス構成・安定化が重い・実費が発生）

一般的なコンテナ Web サービスを模した構成とする。ALB + ECS Fargate + RDS を中心に据える。

- **VPC**（パブリックサブネット + isolated サブネット、NAT なし）
- **Application Load Balancer（ALB）**: プロビジョニングとヘルスチェック安定化に時間がかかる
- **ECS Fargate サービス**: タスクが steady state（healthy）に達するまでの待ちが長い。
  Express Mode の効果が最も出やすい候補
- **RDS インスタンス**: 作成・変更時の安定化が長い
- （任意）**CloudFront ディストリビューション**: エッジ伝播が長い。コストと時間を見て追加を判断する

この構成は「ALB がターゲット（ECS タスク）を healthy と判定するまで待つ」典型的な安定化を含むため、
通常モードと Express モードの完了判定タイミングの差が観測しやすい。

#### ネットワーク方針（計測を安定させるための簡略化）

本検証の目的はデプロイ速度の計測なので、ネットワークはシンプルに保つ。

- **Fargate タスクは public サブネットに配置し、パブリック IP を付与**して IGW 経由で直接 egress する。
  これは、コンテナイメージを public ECR から取得し、CloudWatch Logs へ到達するために外向き通信が
  必要なため。NAT ゲートウェイ経由にするとコールドスタート時の疎通が不安定になり、タスク起動が
  `ResourceInitializationError`（CloudWatch へ到達できない）で失敗することがあったため、
  NAT を廃止して IGW 直結にした（コストも削減できる）。
- **RDS は isolated サブネットに配置**する。RDS は外向き通信が不要なため。
- **ECS デプロイのサーキットブレーカーを有効化**し、タスクが起動に失敗し続けた場合でも
  最大 3 時間ハングせずに数分で失敗が確定するようにする。

一方、軽量セットの **VPC 内 Lambda は isolated サブネットのまま**とする。
この Lambda は外部通信を行わず（目的は ENI 作成・削除の遅さの検証）、egress が不要なため
パブリック化は不要。

軽量セットと重量セットは、1 スタックにまとめても、役割ごとに分割してもよい。
いずれの場合も、通常モード・Express モードそれぞれで 1 回ずつデプロイして比較する。

## プレフィックス設計（名前衝突の回避）

本検証では、同じ CDK コードを「通常モード」と「Express モード」でそれぞれ 1 回ずつデプロイして比較する。
デプロイは並行して行う必要はなく、順番で構わない。

ただし、**比較のために片方をデプロイした後、もう片方をデプロイする際に前のスタックを削除しない**。
そのため 2 つのスタックが同時に存在することになり、スタック名やリソース物理名が衝突すると
デプロイがエラーになる。これを避けることがプレフィックスの唯一の目的である。

デプロイ時にプレフィックスを指定し、スタック名と物理名を要するリソースを分離する。

### プレフィックスの注入方法

- CDK context 経由でプレフィックスを渡す（例: `cdk deploy -c prefix=normal`）。
- `bin/app.ts` で `app.node.tryGetContext('prefix')` を読み取り、既定値（例: `dev`）にフォールバックする。
- プレフィックスはスタック名と、物理名の指定が必要なリソースに前置する。

デプロイ例を以下に示す。

```bash
# 通常モード
cdk deploy -c prefix=normal

# Express モード
cdk deploy -c prefix=express --express
```

### 命名の基本方針

- **物理名は極力指定しない**。CDK はスタック名を含めた一意な名前を自動生成するため、
  物理名を固定しなければリソース衝突は原則発生しない。
- **物理名の指定が必要／グローバル一意なリソースにのみプレフィックスを付与する**。
  該当例は次のとおり。
  - S3 バケット名（グローバル一意）
  - 明示的に名前を付ける IAM ロール・SQS キュー・ロググループなど
- スタック名はプレフィックスとセット種別を組み合わせる。
  例: プレフィックス `express` なら `` `express-light` `` / `` `express-heavy` ``。

### 衝突を避けるための注意

- 2 つのスタックは互いに独立させ、共有リソース（同一名の VPC やロールなど）を作らない。
- グローバル一意なリソース名には、プレフィックスに加えてアカウント ID やリージョンを混ぜて衝突を確実に防ぐ。
- Express モードはロールバックがデフォルト無効のため、失敗しても別プレフィックスで作り直せる利点がある。

## 計測方法

同一の CDK コードに対して、デプロイモードだけを変えて所要時間を計測する。

- 計測対象は `cdk deploy` コマンドの開始から終了までの実時間（実際に待った経過時間）とする。
- 通常モードと Express モードを、それぞれ同一構成・同一リージョンで **1 回ずつ** 実行して比較する。
- 必要に応じて複数回計測し、ばらつきを記録してもよい（必須ではない）。
- CloudFormation のスタックイベントのタイムスタンプも併せて取得し、
  「完了判定の時刻」と「バックグラウンド処理が実際に落ち着く時刻」を突き合わせる。

### 計測結果

計測は `diag/measure.sh`（`diag/single.ts`）で、次の順に通常・Express を比較した。指標は
`cdk deploy` コマンドの開始〜終了までの実時間（実際に待った経過時間）。CDK 合成等で約15秒の固定オーバーヘッドを含む
（SNS/SSM が両モード17秒なのがその目安）。AWS CDK / aws-cdk-lib は最新版（CLI 2.1129.0 / lib 2.260.0）。

#### 1. 単独スタックの新規作成（サービス別）

| サービス | 通常モード | Express モード | 短縮率 |
|----------|-----------|----------------|--------|
| SQS + DLQ | 83秒 | 51秒 | 38.6% |
| VPC 内 Lambda | 191秒 | 160秒 | 16.2% |
| SNS | 17秒 | 17秒 | 0% |
| SSM パラメータ | 17秒 | 17秒 | 0% |
| ALB | 205秒 | 201秒 | 2.0% |
| ECS Fargate | 426秒 | 94秒 | **77.9%** |
| RDS | 446秒 | 425秒 | 4.7% |

- **ECS Fargate が最大**（77.9%）。steady state（healthy）待ちが安定化そのもので、Express が飛ばす。
- SQS（38.6%）・Lambda（16.2%）も短縮。SNS・SSM は安定化待ちが無く 0%（17秒は固定オーバーヘッド）。
- RDS・ALB は実プロビジョニングが支配的で、飛ばせる安定化の割合が小さくほぼ差なし。

#### 2. 全部入り（複合スタック）の新規作成

| 複合スタック | 通常モード | Express モード | 短縮率 |
|--------------|-----------|----------------|--------|
| 新規作成 | 453秒 | 479秒 | ほぼ差なし（誤差） |

一番遅いリソース（RDS 約450秒）に引っ張られ、ECS の短縮も RDS の陰に隠れて全体では効果が消える。

#### 3. 既存の複合スタックに SQS + DLQ を追加する更新

| 複合スタックへの更新 | 通常モード | Express モード | 短縮率 |
|----------------------|-----------|----------------|--------|
| SQS + DLQ を 1 セット追加 | 181秒 | 71秒 | **60.8%** |

RDS・ECS 等は変更なしで待ちが発生せず、操作時間のほぼ全部が SQS の安定化待ち。Express が大きく効く。

#### 結論

Express が短縮するのは「安定化の待ち」だけで、効果はその 1 回のデプロイ操作に占める安定化待ちの
割合に比例する。**大きなスタックの新規作成では効果が埋もれ（RDS が支配的）、既存スタックへの
小さな追加更新では大きく効く（60.8%）**。単独では ECS Fargate で劇的に効く（77.9%）。

## 開発ツール・品質管理

検証コードであっても、可読性と再現性を保つために以下のツールを導入する。

### リント・フォーマット

- **ESLint**: TypeScript の静的解析。`@typescript-eslint` を使用する。
- **Prettier**: コードフォーマットを統一する。ESLint とはフォーマット責務を分離する。
- `npm run lint` / `npm run format` で実行できるよう `package.json` にスクリプトを定義する。

### ユニットテスト

- **Jest** と `aws-cdk-lib/assertions` を用いて、合成した CloudFormation テンプレートを検証する。
- 主な観点は次のとおり。
  - 意図したリソース種別・個数が生成されているか（`Template.resourceCountIs` など）
  - 重要プロパティが期待どおり設定されているか（`Template.hasResourceProperties`）
  - スナップショットテストで意図しない差分が出ていないか
- `npm run test` で実行する。デプロイせずに構成を検証できるため、AWS 課金は発生しない。

### セキュリティ・ベストプラクティス検査（cdk-nag）

- **cdk-nag** を導入し、`AwsSolutionsChecks` を CDK アプリ全体に Aspect として適用する。
- 合成時（`cdk synth`）に、AWS のベストプラクティスに反する設定を自動検出する。
- 検証用に意図的に緩めた設定（例: パブリックアクセス、暗号化の省略）は、
  `NagSuppressions` で理由を明記して抑制し、無効化した項目を記録として残す。
- ユニットテスト内でも cdk-nag の警告・エラーが出ないことを検証し、CI で継続的にチェックできるようにする。

## 前提・コスト・注意点

- 検証には実際の AWS アカウントと課金が発生する。特に RDS・CloudFront・ALB は実費がかかる。
- 検証後は必ずスタックを削除し、リソースの残存がないことを確認する。
- Express Mode はロールバックがデフォルト無効のため、失敗時に中途半端な状態が残り得る。
  検証用アカウントで実施し、本番環境では使わない。
- リージョンは検証コストと再現性のため 1 つに固定する（既定は `ap-northeast-1` を想定）。
- CDK / CLI のバージョンは Express Mode に対応したものを使用する。
